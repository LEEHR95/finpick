# 성능 측정 & 아키텍처 의사결정

FinPick의 상품 검색(RAG)은 **"메모리 numpy → ES 키워드 → ES 의미검색 → +Redis 캐시"** 4개 구조를
**같은 데이터(금융감독원 공시 1,384건)로 측정 비교**한 뒤 설계를 결정했습니다.
기능을 늘리면 한 지표(품질·속도)는 좋아지지만 다른 지표(비용·유지비)는 올라가므로,
**병목을 측정으로 먼저 찾고, 그걸 겨냥한 기능만 붙이는** 원칙으로 접근했습니다.

측정 코드: [`bench_cache.py`](../bench_cache.py) · [`experiment.py`](../experiment.py)

---

## 1. 구조별 응답시간 (동일 질문 5개, ms)

| 질문 | A. 메모리 numpy | B. ES 키워드 | C. ES 의미검색 | D. +캐시(히트) |
|---|--:|--:|--:|--:|
| 안전하게 목돈 굴리기 좋은 예금 | 1,338.9 | 26.7 | 137.6 | 2.8 |
| 이자 높은 적금 | 71.0 | 11.5 | 82.0 | 2.7 |
| 주택 구입 자금 대출 | 73.5 | 52.1 | 100.6 | 2.1 |
| 전세 보증금 마련 대출 | 80.0 | 57.4 | 131.4 | 2.6 |
| 노후 대비 연금 상품 | 70.1 | 53.1 | 84.8 | 2.5 |
| **평균** | **326.7** | **40.2** | **107.3** | **2.5** |

> 응답시간의 병목은 검색 백엔드가 아니라 **질문 임베딩(Upstage API) 호출 — 평균 325ms**.

## 2. Redis 캐시 적용 전 / 후 (의미검색 end-to-end)

| 구성 | 평균 응답시간 | 상대 |
|---|--:|:--|
| 캐시 미적용 | **471.3 ms** | `██████████████████████████` |
| 캐시 미스(첫 호출) | 62.1 ms | `███` |
| 캐시 히트 | **2.4 ms** | `▏` |

**반복 질문 기준 약 195배 향상** (질문당 평균 468.9ms 절약).
챗봇은 같은/유사 질문이 반복되므로 효과가 크고, Upstage API 호출 비용도 절감됩니다.

### 질문별 상세 (ms)

| 질문 | 캐시 미적용 | 캐시 미스(첫호출) | 캐시 히트 |
|---|--:|--:|--:|
| 안전하게 목돈 굴리기 좋은 예금 | 1,874.6 | 99.8 | 1.9 |
| 이자 높은 적금 | 83.7 | 15.9 | 2.1 |
| 주택 구입 자금 대출 | 207.3 | 24.5 | 2.6 |
| 전세 보증금 마련 대출 | 85.2 | 60.8 | 2.8 |
| 노후 대비 연금 상품 | 105.7 | 109.5 | 2.7 |
| **평균** | **471.3** | **62.1** | **2.4** |

## 3. 유지비 · 비용 · 검색 품질

| 항목 | 값 |
|---|---|
| ES 인덱스 크기 (1,384건) | 88.1 MB |
| Redis 사용 메모리 | 1.40 MB |
| 인메모리 매트릭스(A) | 21.6 MB (1384 × 4096) |
| 질문당 임베딩 호출 | A·C 1회 / B 0회 / **D 0~1회(히트 시 0)** |
| 검색 품질 recall@5 | 키워드(BM25) **100%** · 의미검색(kNN) **100%** |

---

## 4. 캐시 정책

| 항목 | 값 | 근거 |
|---|---|---|
| 질문 임베딩 TTL | 7일 | 같은 질문의 의미 벡터는 변하지 않음 |
| 검색결과 TTL | 1시간 | 금리 데이터는 일 단위 갱신 → 신선도·속도 균형 |
| 메모리 정책 | 256MB + allkeys-lru | 한도 초과 시 오래된 캐시부터 자동 삭제 |
| 키 네임스페이스 | `bankrag:종류:해시` | 다른 앱과 키 충돌 방지 |
| 무효화 | 데이터 재적재 시 `flush_kind('search')` | 금리 갱신분 즉시 반영 |

---

## 5. 결론

- **의미검색(C)** 으로 자연어 이해 품질을 확보하되, 병목인 임베딩 호출을 **Redis 캐시(D)** 로 제거해
  반복 질문 응답을 **471ms → 2.4ms (약 195배)** 로 낮췄습니다.
- 데이터가 작을 땐 numpy(A)도 비슷하지만, 확장성(전체 벡터를 메모리에 적재하지 않아도 됨)을 위해 ES를 택했습니다.
- 핵심 교훈: **병목을 먼저 측정하고, 그 지점을 겨냥한 기능을 붙일 때 개선 효과가 가장 크다.**
